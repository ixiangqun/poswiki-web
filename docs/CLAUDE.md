## 一、目录结构与命名规范

```
/POSWIKI/wiki/
  CLAUDE.md          ← 本文件（操作规则，LLM 只读）
  _index.md          ← 主索引（LLM 维护，每次写操作后更新）
  log.md             ← 操作日志（LLM 追加，每次写操作后追加）
  /concepts/         ← 概念词条（中文命名，如：定位理论概述.md）
  /frameworks/       ← 分析框架（中文命名，如：定位决策框架.md）
  /notes/            ← 章节笔记（格式：chNN-主题.md，如：ch05-定位的本质.md）
  /cases/            ← 案例库（英文连字符命名，如：miller-lite.md）
    _cases_index.md  ← 案例索引（LLM 维护）
```

**命名规则**：
- `/concepts/` 和 `/frameworks/`：使用中文名，如 `定位理论概述.md`、`竞争地图.md`
- `/notes/`：`ch` + 两位章节号 + `-` + 中文主题，如 `ch07-进入心智.md`
- `/cases/`：全小写英文，单词间用连字符，如 `miller-lite.md`、`xerox-computer.md`

## 二、新书拆解工作流

**触发方式**：用户说"处理《书名》"或"拆解新书"

### Step 1：提取 EPUB 内容

使用 `poswiki-process-epub` skill 提取指定章节文本。若用户未指定 EPUB 路径，在以下位置搜索：
- `/Users/lixiangqun/Work/AI Positioning/Raw/`
- `/Users/lixiangqun/Downloads/`

### Step 2：内部提炼（每章）

对每章在内部提炼（不写文件）：
- 核心主张（3-5条）
- 关键案例（含量化数据）
- 可引用金句
- 可更新的现有概念（对照 `/concepts/` 文件名）

### Step 3：路由决策（必须暂停等待确认）

向用户汇报：

> **路由决策：**
> - 将新建章节笔记：`chNN-主题.md`
> - 将更新概念词条：`概念名.md`（理由：XXX）
> - 将新建案例：`案例名.md`（触发条件：XXX）
>
> 请确认后继续。

**决策规则：**
- 每章必须新建一个章节笔记（`/notes/`）
- 章节中出现"**公司/产品名 + 量化结果**（百分比、金额、排名变化）"→ 同步创建 `/cases/` 条目
- 出现现有概念无法覆盖的核心理论 → 新建概念词条
- 出现可扩展现有概念的新内容（≥2条） → 更新该概念词条

### Step 4：写入文件

按此顺序写入：
1. 新建或更新 `/notes/chNN-*.md`
2. 新建或更新 `/concepts/*.md`（追加，不覆盖现有内容）
3. 新建或更新 `/cases/*.md`（若触发条件）
4. 更新 `cases/_cases_index.md`（若新建案例）
5. 更新 `_index.md` 统计数字
6. 追加 `log.md`（每次必须执行）

## 三、案例触发规则

**触发条件（满足任意一项）**：
- 笔记中出现公司名 + 数字变化（如：市场份额、营收、排名）
- 笔记中有明确的"定位前 → 定位后"叙述
- 里斯·特劳特以该案例作为核心论证依据

**案例文件模板**：

```markdown
# [公司名] — [一句话概括定位动作]

> 行业：[行业]
> 定位动作：[核心战略动作]
> 来源章节：[第N章：主题](../notes/chNN-主题.md)

---

## 问题诊断

[定位前的状况，遭遇了什么挑战]

---

## 定位方案

[采取了什么定位策略，核心概念是什么]

---

## 执行结果

[量化数据：数字、比例、时间线]

---

## 理论注解

- [概念名](../concepts/概念名.md) — [与本案例的关联]
```

**案例文件命名**：使用英文连字符，如 `avis-car-rental.md`、`fedex-overnight.md`

## 四、概念词条更新规则

| 情况 | 操作 |
|------|------|
| 新章节补充现有概念内容 | 在相关 `##` 章节末尾追加 |
| 新角度无对应章节 | 在"## 相关概念"之前新建 `##` 章节 |
| 与现有内容矛盾 | 保留双方，注明章节来源，不删除旧内容 |
| 纯重复内容 | 跳过，不写入 |

**概念文件格式**（新建时）：

```markdown
# [概念名]

> 来源：《定位》里斯·特劳特

---

## 核心命题

[2-3句话]

---

## [章节一]

[内容]

---

## 相关概念

- [概念名](概念名.md) — [关联说明]
```

## 五、`_index.md` 统计更新规则

每次写操作后，同步更新 `_index.md` 顶部统计表：

| 分类 | 统计方法 |
|------|---------|
| 概念词条 | 统计 `/concepts/` 下 `.md` 文件数 |
| 分析框架 | 统计 `/frameworks/` 下 `.md` 文件数 |
| 章节笔记 | 统计 `/notes/` 下 `.md` 文件数 |
| 案例库 | 统计 `/cases/` 下 `.md` 文件数（排除 `_cases_index.md`） |

同时更新"最近更新"日期（使用当前日期）和"总词条"数字。

## 六、`log.md` 追加规则

**每次写操作必须追加 `log.md`**，格式如下：

```markdown
## [YYYY-MM-DD] [操作类型] | [简短描述]
- [具体操作1]
- [具体操作2]
```

**操作类型**：
- `ingest` — 处理新书或新章节
- `new-concept` — 新建概念词条
- `update-concept` — 更新概念词条
- `new-case` — 新建案例文件
- `new-framework` — 新建框架文件
- `lint` — 执行健康检查
- `query-as-note` — 保存合成笔记
- `maintenance` — 索引、日志等维护操作

## 七、Lint 操作

**触发方式**：用户说 "lint"、"健康检查"或"知识库检查"

**检查项**：

| 编号 | 名称 | 检查内容 |
|------|------|---------|
| L1 | 孤儿概念 | `/concepts/` 中未被 `_index.md` 引用的文件 |
| L2 | 孤儿案例 | `/cases/` 中未被 `_cases_index.md` 引用的文件（排除 `_cases_index.md` 本身） |
| L3 | 无来源引用 | 概念词条中出现量化数据但无章节来源引用 |
| L4 | 空案例链接 | `_cases_index.md` 中"链接"列为"—"的行 |
| L5 | 章节笔记孤立 | `/notes/` 中"与其他章节的联系"为空或仅有一个链接的笔记 |

**输出格式**：
```
## Lint 报告 [YYYY-MM-DD]

### L1 孤儿概念
- [列出文件名]

### L2 孤儿案例
- [列出文件名]

...（无问题则输出 ✅ 无问题）

**建议行动**：[优先修复项]
```

执行 lint 后，追加 `log.md`（操作类型：`lint`）。

## 八、Query-as-Note 模式

当回答需要跨越 **2 个以上概念文件** 且回答超过 **300 字**时，回答结束后询问：

> "这个回答综合了 [概念A]、[概念B]、[概念C] 的内容。是否存为合成笔记？"

**合成笔记存储位置**：`/concepts/` 目录

**合成笔记模板**：

```markdown
# [合成主题名]

> 类型：合成笔记（Query-as-Note）
> 来源综合：[概念A](概念A.md) · [概念B](概念B.md) · [概念C](概念C.md)
> 生成日期：YYYY-MM-DD

---

[回答正文]

---

## 参考来源

- [概念A](概念A.md) — [关联说明]
- [概念B](概念B.md) — [关联说明]
```

## 九、禁止事项

- **不修改** `/notes/` 中任何已有章节笔记内容（只追加或新建）
- **不删除** 任何现有概念词条内容（只追加）
- **不修改** 概念词条的标题（`#` 行）或已有章节标题（`##` 行）
- **不在** 概念词条中添加 YAML frontmatter
- **不修改** CLAUDE.md 本身（本文件）
- **不修改** `_cases_index.md` 和 `_index.md` 的表头结构，只追加数据行
